# 2026-07-13 ATO v0.2.0 Productization Design

## Goal

Turn ATO 0.1.0 from a repository-bound technical preview into an installable, truthful, and auditable multi-agent orchestration product. A clean machine must be able to install the Python core from PyPI and the CLI/MCP adapters from npm, run `ato doctor`, start isolated tasks, execute independent subtasks concurrently, pause on restricted tools, resume from an explicit approval, and inspect task-scoped status and audit data. The release target is `v0.2.0` because the broken `src.*` Python import surface is replaced by the real public package `ato_core`.

## Non-Goals

- No Web UI or TUI dashboard.
- No Docker platform, hosted control plane, or remote multi-machine workers.
- No database service or shared multi-user tenancy.
- No new LLM providers beyond the existing Claude CLI, Anthropic, OpenAI, and Ollama adapters.
- No compatibility alias for `src.*`; that import path was never usable as an installed public package.
- No silent automatic installation of Python from npm and no package-manager scripts that mutate the user's global Python environment.
- No claim that a denied or failed task completed successfully.

## Source of Truth and Owner Layer

### Python core

`ato_core` is the single owner of:

- Task lifecycle and status transitions.
- Task decomposition validation.
- LangGraph state, checkpointing, concurrency reducers, and resume behavior.
- Tool definitions, tool argument validation, permission classification, execution, and audit events.
- Approval requests and approval decisions.
- Role definitions and role schema resources.
- Task result and team-memory semantics.
- Stable bridge request and response schemas.

The installable source layout becomes:

```text
packages/core/
├── pyproject.toml
├── src/
│   └── ato_core/
│       ├── bridge/
│       ├── runtime/
│       ├── orchestrator/
│       ├── models/
│       ├── tools/
│       ├── memory/
│       └── resources/roles/
└── tests/
```

Role files and `role.schema.json` are package resources loaded through `importlib.resources`. Repository checkouts and installed wheels use the same loader.

### TypeScript adapters

- `@spacesky-cell/ato-shared` owns process discovery, the JSON/JSONL transport, timeout handling, and typed bridge envelopes.
- `@spacesky-cell/ato-cli` owns terminal presentation and interactive input only. It exports `main(argv)` without executing on import; its bin module invokes that function. It does not decide task status, tool permission, or approval truth.
- `@spacesky-cell/ato-mcp-server` maps MCP requests to bridge commands and bridge responses back to MCP content. It exports `main()` without executing on import; its bin module invokes that function. It does not construct Python programs or infer business state.
- `@spacesky-cell/agent-team-orchestrator` is the user-facing npm entry package. It pins matching-version CLI and MCP runtime dependencies and exposes its own thin `ato` and `ato-mcp` bin wrappers that call those exported entry functions.

## Proposed Design

### 1. Distribution and discovery

The Python distribution is named `ato-core`; the import package is `ato_core`. The wheel includes role YAML files, the role schema, package metadata, LICENSE, and package documentation. `python -m ato_core.bridge doctor` must work outside a repository checkout.

The supported installation is:

```bash
pip install ato-core==0.2.0
npm install -g @spacesky-cell/agent-team-orchestrator@0.2.0
ato doctor
```

Node discovers Python in this order:

1. Explicit `ATO_PYTHON` executable.
2. `.venv` or `venv` under the requested project root.
3. `python` on Windows; `python3` then `python` on Unix.

Every candidate is verified by running a bounded import probe for `ato_core`. A present executable without `ato_core` is rejected with `CORE_NOT_INSTALLED`; discovery continues to the next candidate. Error output lists attempted executables but does not expose environment secrets.

### 2. Stable bridge protocol

Node no longer writes temporary Python source files. It invokes stable module commands:

```text
python -m ato_core.bridge doctor
python -m ato_core.bridge roles-list
python -m ato_core.bridge task-start
python -m ato_core.bridge task-run
python -m ato_core.bridge task-status
python -m ato_core.bridge task-approve
python -m ato_core.bridge task-audit
python -m ato_core.bridge task-list
python -m ato_core.bridge memory-query
python -m ato_core.bridge memory-summary
```

Request payloads are JSON objects on stdin. One-shot commands return exactly one JSON response on stdout:

```json
{"ok":true,"data":{}}
```

Failures use a stable code and a user-safe message:

```json
{"ok":false,"code":"CORE_NOT_INSTALLED","message":"ato-core is not installed for the selected Python executable","details":{"python":"python"}}
```

Foreground `task-run` uses JSONL events for progress and approval requests. The CLI renders events and sends approval responses without changing their meaning. Human-readable logs go to stderr so stdout remains machine-readable.

### 3. Task identity and persistence

Task IDs are UUID-based and generated by `ato_core`. A configurable output root defaults to `./ato-output`; every task has an isolated directory:

```text
ato-output/tasks/<task-id>/
├── task.json
├── result.json
├── checkpoints.db
├── approvals.jsonl
└── tool-audit.jsonl
```

`task.json` is the current task summary and is written atomically with a temporary sibling plus `os.replace`. `result.json` is written only for terminal states. JSONL audit files are append-only and written by the task coordinator under a process-local lock. LangGraph checkpoint state remains in the task's `checkpoints.db`.

The task state machine is:

```text
queued -> decomposing -> running -> completed
                           |  |  |
                           |  |  +-> failed
                           |  +----> blocked
                           +-------> waiting_approval -> running
                                                     -> blocked
```

Allowed terminal states are `completed`, `blocked`, and `failed`. `waiting_approval` is durable: the process may exit while the checkpoint and pending request remain resumable.

`task.json` records `task_id`, `status`, `created_at`, `updated_at`, `project_root`, `output_dir`, progress counts, active approval request, last error code, worker PID, and heartbeat timestamp. Status reporting must use these persisted fields. When a running worker PID is gone and its heartbeat is stale, `task-status` returns a structured unavailable/failure reason rather than reporting healthy progress.

`WorkerLauncher` is the only process-management boundary. It starts background workers with an argument list rather than a shell string, closes inherited handles, reports spawn failure before `task-start` returns success, and implements platform-specific PID liveness checks behind one interface. The coordinator refreshes the heartbeat at bounded graph transitions rather than through an unbounded polling thread.

### 4. Decomposition validation and parallel execution

Before graph execution, `ato_core` validates:

- At least one subtask exists.
- Subtask IDs are non-empty and unique.
- Every referenced role exists.
- Every dependency references an existing subtask.
- No subtask depends on itself.
- The dependency graph is acyclic.

Parallel branches do not receive and return a mutable copy of the complete `TeamState`. Each execution branch returns a `SubtaskExecutionResult` delta containing the subtask ID, terminal subtask status, artifact, messages, and audit events. Explicit reducers merge only those delta fields. Immutable task identity and configuration remain coordinator-owned context.

The graph must support at least two ready subtasks in one superstep, dependency fan-in, one-branch failure, and blocked dependency propagation without concurrent state-update errors or false completion.

### 5. Tool schemas and execution boundary

Tool JSON schemas are the source for both Claude CLI structured responses and LangChain tool binding. Required properties use Pydantic required sentinels; defaults and descriptions are represented separately. A schema round-trip test verifies required arrays, enums, numeric values, booleans, and unknown-field rejection.

Every tool receives an immutable execution context containing `task_id`, `subtask_id`, `project_root`, and allowed directories. File and subprocess paths resolve against `project_root`. `GitCommitTool` runs all git commands with `cwd=project_root` and verifies that the discovered repository root is the requested project or a child repository explicitly selected by the task.

`safe_mode` is not model-controlled. Command safety and approval are runtime policy. Command, test, write, delete, and git operations have explicit timeouts and bounded captured output. Tool errors return structured codes and are always audited.

### 6. Real approval workflow

Tool permission has two separate owners:

- `ToolPolicy` classifies a request as automatically allowed, approval required, or denied.
- `ApprovalProvider` obtains or resumes the decision without changing the classification.

Read-only tools are automatically allowed. Mutating filesystem operations, command execution, test execution, and git commits require approval.

Providers are:

- `InteractiveApprovalProvider`: used by foreground CLI execution; renders the exact tool, summarized arguments, project root, and risk class.
- `InterruptApprovalProvider`: used by MCP/background execution; creates a request, checkpoints the graph, writes `waiting_approval`, and exits cleanly.
- `AutoApprovalProvider`: enabled only by `ATO_AUTO_APPROVE_TOOLS=1` for trusted local debugging. `doctor` reports this mode prominently.

Each approval request has a unique `request_id`. `approve_step` requires `taskId`, `requestId`, and `approved`. It verifies that the request is current and unresolved, appends the decision to `approvals.jsonl`, and resumes the checkpoint through a new worker. Duplicate decisions are idempotent; mismatched or stale request IDs return `APPROVAL_NOT_PENDING`.

An explicit rejection marks the affected subtask and overall task `blocked`. It never reports completion. An execution failure after approval marks the tool audit event and task failure separately from the approval decision.

### 7. CLI and MCP behavior

CLI commands remain `run`, `roles`, `status`, `audit`, `tasks`, `memory`, `doctor`, and `init`. Their behavior changes to the stable bridge. `run` is foreground and interactive by default. `--detach` uses `task-start` and immediately prints the task ID. Status and audit commands require a task ID unless exactly one task exists under the selected output root.

MCP tools are:

- `create_team_task`: starts a background task and returns `taskId`, initial status, and task directory immediately.
- `get_task_status`: returns persisted task state by task ID.
- `get_task_audit`: returns task-scoped counts and recent events by task ID.
- `approve_step`: submits a decision for a specific pending request and resumes execution.
- `list_available_roles`, `list_incomplete_tasks`, `query_team_memory`, `get_memory_summary`, and `self_check`: delegate to equivalent bridge commands.

MCP errors set `isError: true` and include the stable bridge code. CLI errors return non-zero exit codes. Neither adapter catches an error and substitutes a success message.

### 8. Team memory truth

Team memory records architecture decisions only when the orchestrator receives an explicit structured decision artifact. Code-change records come from successful write/delete tool audit events and verified git diffs, not regex matches over an LLM narrative. When no verified change exists, memory records no code change.

ChromaDB remains optional. When unavailable, status reports semantic search as unavailable and uses SQLite recent-record retrieval without claiming semantic search is active.

### 9. Error handling and bounds

- Bridge requests validate input before touching task state.
- Default one-shot bridge timeout is 300 seconds; task execution uses a configurable bounded task timeout with a 1,800-second default.
- Subprocess stdout and stderr are truncated to a configured maximum before persistence or transport.
- Worker spawn failure marks the task failed atomically with `WORKER_START_FAILED`.
- Invalid or corrupt task files return `TASK_STATE_CORRUPT`; they are not replaced with empty success states.
- Missing tasks return `TASK_NOT_FOUND`.
- Invalid dependency graphs return `INVALID_DECOMPOSITION` with the failing IDs.
- Provider authentication and CLI availability errors are distinct from task/tool failures.
- Secrets are redacted by key name and known credential patterns before audit persistence.
- Project path containment is enforced in `ato_core`; Node normalization is advisory only.

### 10. CI, documentation, and release

CI runs on pull requests to `main`, pushes to `main`, and release tags. Ruff, Black, MyPy, ESLint, TypeScript compilation, Python tests, TypeScript tests, package builds, and cold-install smoke tests are strict jobs. No quality command uses `|| true`.

Documentation updated in the same release:

- README.md and README_CN.md share the same supported installation and capability claims.
- QUICKSTART starts from a clean environment and completes `ato doctor` plus a no-LLM role/status smoke.
- MCP_GUIDE documents asynchronous task IDs and approval request IDs.
- architecture.md documents the owner boundary and bridge protocol.
- CONTRIBUTING.md uses pnpm, the actual Python dev extra, targeted staging, and exact quality commands.
- SECURITY.md contains a real reporting path and accurately describes command approval and path containment.
- Package metadata contains repository, homepage, bugs, author, license, keywords, and supported runtime versions.
- README includes CI/npm/PyPI/version/license badges, one verified terminal demonstration, task/audit artifacts, honest limitations, and a concise comparison with raw LangGraph, CrewAI, and AutoGen.

Release sequence:

1. Complete implementation commits and pass all local gates.
2. Build and inspect Python wheel/sdist and all npm tarballs.
3. Set all public package versions to `0.2.0`, update CHANGELOG, and create the release commit.
4. Fast-forward or merge the verified branch into local `main` without discarding existing commits; rerun release gates on the exact release commit.
5. Push `main` and wait for its required CI jobs to pass.
6. Create and push tag `v0.2.0`; wait for tag CI to rebuild and verify the same source.
7. Build final artifacts from a clean checkout of `v0.2.0`, install the local wheel/tarballs in clean environments, and write a release manifest containing artifact filenames and SHA-256 digests.
8. Publish `ato-core` to PyPI, then publish npm packages in dependency order: shared, CLI, MCP, root entry package. Published bytes must match the release manifest.
9. Install from public registries in a clean venv and temporary npm project; run `ato doctor`, CLI smoke, and MCP startup smoke.
10. Create a non-draft GitHub Release from the verified changelog, attach the release manifest, and add repository topics for `multi-agent`, `langgraph`, `mcp`, `claude-code`, `agent-orchestration`, `python`, and `typescript`.

Publishing stops before any registry mutation if PyPI or npm authentication is unavailable, the package name is no longer available, the registry version already exists with different contents, local artifact installation fails, or source/tag CI is not green. Credentials are provided through temporary environment/configuration only and are never written to the repository or command output. If a registry fails after an earlier artifact was published, the workflow records the partial state, keeps the original artifact bytes immutable, and retries only the missing package; it never rebuilds or overwrites an already published version.

## Alternatives Considered

### Repository-only installation

This would remove npm/PyPI distribution complexity, but it leaves onboarding dependent on a source checkout and does not satisfy the goal of attracting users through a reliable installation path.

### npm-only rewrite

Rewriting orchestration in TypeScript would create a single package-manager experience but discards the working LangGraph/Python core and substantially increases scope and regression risk. It is not justified for v0.2.0.

### Bundle or auto-install Python from npm

Bundling a Python runtime or invoking pip from npm lifecycle scripts creates large artifacts, platform-specific failures, and surprising global environment mutations. Explicit dual installation with a strong doctor command is more predictable.

## Migration or Compatibility Policy

- `src.*` imports are removed and not aliased. All documentation and internal scripts use `ato_core.*`.
- Existing repository-local task outputs are not migrated automatically because 0.1.0 used one shared `result.json` without enough identity data for a lossless task split.
- Existing npm package names remain, but all public packages move together to 0.2.0.
- Existing command names remain where their semantics are truthful; `approve_step` gains required `requestId` and real persistence.
- `ATO_AUTO_APPROVE_TOOLS=1` remains supported only as an explicitly unsafe local debugging mode.

## Acceptance Checks

### Python package and core

- `python -m pytest packages/core/tests -v --cov=ato_core --cov-report=term-missing`
- `python -m black --check packages/core/src packages/core/tests`
- `python -m ruff check packages/core/src packages/core/tests`
- `python -m mypy packages/core/src`
- Build wheel and sdist in an isolated environment.
- Install the wheel into a clean venv and verify `import ato_core`, bundled role loading, bridge doctor, and bridge role listing outside the repository.

### Orchestration and permissions

- Two independent ready subtasks execute in one graph superstep without `InvalidUpdateError`.
- Dependency fan-in waits for all dependencies and merges each artifact exactly once.
- Duplicate, missing, self, and cyclic dependencies fail before graph execution.
- Interactive approval allow and deny cases persist correct audit decisions.
- MCP/background approval pauses, survives process exit, rejects stale request IDs, and resumes once.
- Denial produces `blocked`; tool execution errors produce `failed`.
- Two simultaneous tasks use different directories, checkpoints, results, and audit logs.
- File, command, and git tools allow project-root paths and reject paths outside it.
- Git smoke verifies the commit is created only in the requested temporary repository.
- Tool schema tests verify required fields are truly required.

### TypeScript adapters and packages

- `pnpm run build`
- `pnpm exec vitest run --coverage --passWithNoTests`
- `pnpm run lint`
- Bridge parser tests cover success, structured failure, malformed JSON, timeout, non-zero exit, and stderr noise.
- MCP tests cover asynchronous create, status, audit, approval allow/deny, missing task, and corrupt state.
- Inspect each `npm pack --dry-run --json` result for bins, runtime files, README, LICENSE, and resolved workspace versions.

### Public release

- Install `ato-core==0.2.0` from PyPI in a clean venv.
- Install `@spacesky-cell/agent-team-orchestrator@0.2.0` from npm in a clean temporary project.
- Run `ato doctor`, `ato roles`, `ato --version`, and start `ato-mcp` from the installed packages.
- Confirm GitHub default `main` contains the release commit and all CI checks pass.
- Confirm npm and PyPI metadata point to the GitHub repository.
- Confirm GitHub Release `v0.2.0` exists and is not draft or prerelease.

## Risks

- `blocking`: Registry authentication may be unavailable. Mitigation: complete all build and dry-run checks, then stop before publishing and request credentials through a temporary secure channel.
- `blocking`: The PyPI name may become unavailable before release. Mitigation: re-query immediately before versioning; stop and select an explicit replacement name rather than publishing under an unrelated package.
- `design_risk`: Durable LangGraph interrupt/resume behavior may differ across installed LangGraph versions. Mitigation: pin a tested compatible range and add process-exit/resume integration tests before changing adapter APIs.
- `design_risk`: Cross-platform detached workers and PID health checks differ on Windows and Unix. Mitigation: isolate process management behind `WorkerLauncher` and test Windows plus Linux CI paths.
- `design_risk`: Removing `src.*` is breaking. Mitigation: use the 0.2.0 minor boundary while pre-1.0, document the migration, and do not carry a polluted alias.
- `design_risk`: Strict MyPy currently exposes substantial debt. Mitigation: type the owner-layer interfaces first, remove ignored failures, and keep the release blocked until the configured strict command passes.
- `recorded_debt`: Cancellation, retry policies, remote workers, UI dashboards, and multi-user storage remain outside v0.2.0.
- `recorded_debt`: ChromaDB semantic search quality is not benchmarked in this release; availability and fallback truthfulness are verified instead.

## Open Questions

No design question blocks implementation. PyPI and npm credentials are operational release inputs and are requested only after all non-mutating release gates pass.
